## 三、正误评估

### ✅ 做得对的（核心架构决策）

1. **纯本地引擎** (v2.9.9.10-v2.9.13)：从依赖LLM到零API依赖，是项目最重要的架构决策。不仅降低了成本/延迟，也让离线场景和隐私场景成为可能。

2. **管道化架构** (v2.9.10-14)：四管道聚合（画像+场景+粒子+织线） → 3D合成 → pipe_insights，数据流清晰可追溯。

3. **五维画像+知识图谱** (v2.9.14-23)：从自由文本到结构化JSON，fact_tags 自动提取→分类→回填→持久化的全链路。

4. **多路召回搜索** (v2.9.9.8-10)：影子沙+FTS5+IDX+TFIDF 四路并发，各有侧重。

5. **自愈能力** (v2.9.41-42)：新用户0损坏 + 老用户自动修复 + 死锁解除，工程质量的标志。

6. **except精确化** (v2.9.9系列)：bare except→具体异常类型，减少掩盖bug。

7. **并发安全** (v2.9.9.13 + TOCTOU)：RLock + 原子赋值。

8. **PyPI标准化发布** (v2.10.1-5)：让项目可被 `pip install`。

### ⚠️ 有争议的（值得讨论）

1. **三块式→两段式回退** (v2.9.33→v2.9.34)：设计了三块式 150-250t 注入方案（对标 Mem0），但最终实施的两段式只有 ~60t。**信息量是否不足？** 尤其是"记忆候选"块被砍掉后，LLM失去了"当前可搜索到什么"的上下文引导。

2. **v2.9.9.8 高斯排序的过度迭代**：约25个 commit 在高斯位置、反密度中心、长度补偿之间反复摆动。最终方案（统一高斯+IDF标签权重）是对的，但路径太曲折。**根本原因：缺乏离线评估数据集（benchmark）来指导排序调参，纯靠试错。**

3. **版本号体系混乱**：v2.9.42、v2.9.9.10、v2.9.9.13 等四级版本号，以及 v2.9.9.10→v2.10.1 的跳跃，没有语义版本控制的规律。**用户/开发者难以理解版本含义。**

4. **知识图谱标签首次激活又撤回** (v2.9.9.5)：`Reapply "v2.9.9.5: 知识图谱标签激活"` → `Revert "..."` → 后面又重新实现。**缺乏测试覆盖导致首次上线不自信。**

5. **两段式注入里砍掉了 queue_prefetch 的预热搜索词** (v2.9.32→v2.9.34)：v2.9.32 激活了 queue_prefetch（语义扩展预热搜索词），这是一个性能优化的好方案，但在后来的两段式注入里似乎被边缘化了。

### ❌ 可能做错的

1. **distill()先删后补** (v2.9.9.10→v2.9.35)：v2.9.9.10 删除了 distill() 声称"实时管道已完全替代"，但 v2.9.35 又出现 "P0修复 — local_distill替代崩溃的distill()"，说明删除决策过于仓促，蒸馏功能仍有需求。

2. **sand_density 反复调整** (v2.9.9.8→v2.9.9.12)：在反密度→高斯→线性→ratio 之间多次反复。最终 ratio 是对的，但前期的"长度补偿+15%""撤回entities密度"等尝试浪费时间。

3. **测试覆盖严重不足**：v2.9.9.1 才加 `test_smoke.py`（6项零依赖），而且是最小冒烟测试。v2.9.9.5 的 revert/reapply 说明缺乏回归测试。排序迭代没有 benchmark 数据集。

4. **README.txt 含隐私信息** (v2.9.9.7)：`删除README.txt(含隐私信息)` — 说明之前有隐私泄露到仓库中。

5. **MCP inputSchema 补丁滞后** (MR#27)：19个MCP工具的inputSchema在v2.9.9才补上，说明前期MCP工具实现不规范。

## 四、可讨论的优化点

### 🔴 高优先级

1. **注入策略重新评估**
   - 当前两段式 ~60t 是否足够？turn_injection_plan.md 设计的三块式 150-250t 是否应该重新考虑？
   - 建议：AB 测试两段式 vs 三块式，对比 LLM 的 sandglass_search 调用准确率和最终回答质量。
   - 关键取舍：token 成本 vs 信息丰度。如果 60t 确实足够，需要数据证明。

2. **建立搜索排序 benchmark**
   - v2.9.9.8 的 25 次高斯迭代说明缺乏离线评估能力。
   - 建议：做 50-100 条标注 query，记录标准答案（哪些沙子应该被召回），建立 precision/recall/NDCG 评估流程。
   - 每次排序改动前跑 benchmark，替代试错。

3. **版本号语义化**
   - 从四级版本号中选一套：要么 SemVer（MAJOR.MINOR.PATCH），要么 CalVer。
   - v2.10.5 之后建议 v2.11.0 → v2.11.1 → ... 或 v3.0.0。

4. **补测试**
   - test_smoke.py 只有 6 项，远不够。
   - 至少补充：搜索排序回归测试、注入输出格式测试、知识图谱增删改测试、并发安全测试。

### 🟡 中优先级

5. **搜索管道模块化**
   - 当前 sandglass_think.py 610行改动，sandglass_vault.py 413行改动，文件过于庞大。
   - 建议：搜索排序逻辑独立成 `ranking.py`，画像构建独立成 `profiling.py`。

6. **queue_prefetch 恢复或明确废弃**
   - v2.9.32 的预热搜索词如果被边缘化，应明确原因并记录决策。
   - 如果有价值，应在两段式注入中恢复。

7. **distill/local_distill 统一**
   - 当前 local_distill 和管道洞察功能重叠，评估是否合并。

8. **铁律因子功能落地**
   - v2.10.5 标题提到"铁律因子"，但具体改动需要检查是否真正落地到注入/搜索链路中。

### 🟢 低优先级

9. **README 版本历程表可自动化**
   - 当前手动维护，可以写脚本从 git tag 自动生成。

10. **MCP工具 schema 自动校验**
    - MR#27 补了 19 个工具的 inputSchema，应加 CI 自动校验 schema 合规性。

11. **移除 build/lib 历史遗留**
    - git diff 显示大量 `build/lib/` 下的文件（sandglass_think.py, persona_l3.py 等），这些是打包产物，不应在源码仓库中。

## 六、结论

**V2.9→V2.10.5 是一个从"LLM驱动的原型"到"纯本地的生产级产品"的蜕变过程。** 核心架构决策（纯本地引擎、管道化、五维画像+知识图谱、多路召回）都是正确的。最大的问题在于：

1. **搜索排序缺乏数据驱动**（试错太多）
2. **注入策略可能偏保守**（三块式→两段式，60t可能不够）
3. **测试覆盖严重不足**（导致 revert/reapply 反复）
4. **版本号缺乏规范**（四级版本号+跳跃）

下一步优化建议按优先级排序：**补benchmark → 评估注入策略 → 补测试 → 版本规范化**。
